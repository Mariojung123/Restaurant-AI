Why We Build

Motivation

Built from firsthand experience running a restaurant. The pain points below are real, observed daily by owners who manage inventory manually.

👁️
"There / Not There" Thinking

Owners scan the fridge and move on if something is present — even if it's half-gone. The blind spot is "about to run out."

😓
Counting After Closing Is Unrealistic

After a full shift, no one wants to count every ingredient. End-of-day inventory checks are consistently skipped or rushed.

📋
Manual Delivery Entry

Writing down delivery items by hand and transferring to spreadsheets leads to typos, lost records, and missed items.

🎲
Ordering by Gut Feel

"Looks like about 3 days of chicken left" — intuition-based ordering leads to constant over-buying or running out.

Core Concept

The Idea

"Snap a photo of the delivery invoice and the closing receipt — the AI tells you when you'll run out of everything."

A conversational AI that feels like messaging a knowledgeable manager. Owners ask questions in plain language and get data-driven answers. Number of manual data entries per day: zero.

  • How humans see inventory — present / absent
  • How the AI sees inventory — present / running low / absent
  • Natural language units — "half a bell pepper", "a tablespoon of garlic" — understood as-is

System Design

Architecture

Strict three-layer separation enforced across the entire codebase. Every request flows through the same pipeline — no business logic in routers, no HTTP concerns in services.

Frontend
pages/Chat.jsx pages/Dashboard.jsx pages/Recipe.jsx pages/Invoice.jsx pages/Receipt.jsx
↓ fetch via api/ client modules
Routers
chat.py recipe.py invoice.py receipt.py inventory.py vision.py
↓ validate input → call services → return HTTP response
Services
claude.py chat_context.py pending_recipe.py recipe_svc.py invoice.py receipt.py vision_common.py prediction.py fuzzy_match.py ingredient.py chat_history.py
↓ business logic · Claude API calls · DB writes
Models
database.py
  • Chat pipeline — chat_context.py builds a DB snapshot for the prompt → claude.py calls Claude API → pending_recipe.py manages two-step recipe confirmation state → chat_history.py persists turns
  • Vision pipeline — vision_common.py encodes images → shared Claude Vision call → invoice.py or receipt.py parses response and writes DB
  • Fuzzy matching — fuzzy_match.py normalizes extracted ingredient names against DB entries; used by both invoice and receipt parsers
  • Prediction pipeline — prediction.py joins sales_logs × recipe_ingredients to compute daily consumption, then projects depletion dates with a 14-day lookback

MVP Features

What Gets Built

01

Natural Language Chat Q&A

Claude answers operational questions in plain English based on the restaurant's actual data. "What sold best this week?", "When does the chicken breast run out?" — all handled conversationally. Multi-turn context preserved per session. Recipe registration can also be triggered mid-chat — owner can modify details before final confirmation without restarting the flow.

Claude Text API
02

Recipe Registration — Conversational + Structured Picker

Owner says "add chicken salad" → Claude proposes a standard recipe → owner reviews via a structured ingredient picker UI, editing quantities and units inline before confirming. Mid-flow corrections supported without restarting. Accepts natural units throughout: "half a bell pepper", "a splash of olive oil", "two cloves of garlic."

Chat input / natural language Claude proposes recipe Structured ingredient picker Owner confirms → DB saved
Claude Text API
03

Delivery Invoice → Auto Stock Update

Upload one photo of the delivery invoice → Claude Vision parses items and quantities → preview screen shows extracted data for review → one-tap confirm updates stock. Fuzzy matching resolves name variations (e.g., "chicken breast (bone-in)" → existing "Chicken Breast" entry). On parse failure, the system clearly identifies which items it couldn't read.

Photo upload Claude Vision parse Preview & review Confirm → stock updated
Claude Vision API
04

Closing Receipt → Auto Sales Log

Upload a POS daily report or closing receipt photo → Claude Vision extracts per-item sales counts → preview screen shows parsed results → confirm to log. No manual counting required at end of shift. Fuzzy matching links extracted dish names to registered recipes. Unit conversion (kg/g, L/mL) applied before stock deduction.

Photo upload Claude Vision parse Preview & review Confirm → sales logged
Claude Vision API
05

Inventory Depletion Prediction Alert

Combines current stock + 14-day sales history + recipe ingredient consumption to calculate when each item will run out. Dashboard shows depletion date, daily consumption rate, and reorder threshold alert. Proactive, not reactive: "chicken breast runs out tomorrow evening" before it happens. Per-ingredient forecast also available.

Prediction Engine

Tech Stack

Technology Choices

Frontend
React + Vite + Tailwind
PWA-ready via vite-plugin-pwa. Installable on mobile without an app store.
Backend
FastAPI (Python)
Official Anthropic Python SDK. Rich image processing libraries. Native async support for concurrent Vision + chat calls.
Database
PostgreSQL + SQLAlchemy 2.0
Free PostgreSQL on Railway/Render. Clean ORM layer — future=True for 2.0 style. pool_pre_ping for connection health.
AI
Claude API (Sonnet 4.6)
Text + Vision in a single model. Natural language unit parsing, recipe suggestion, invoice and receipt OCR. Model selected per task complexity: Haiku for lookups, Sonnet for general use.

Free tier for React/Vite. Auto-deploy from GitHub.
Deploy · Backend

FastAPI + PostgreSQL on free tier. Simple environment variable management.

Database

DB Schema

ingredients
Raw ingredient catalog — stock level, reorder threshold, base unit
id PK name unit current_stock reorder_threshold notes created_at updated_at
recipes
Menu item catalog — name, description, price per serving
id PK name description price created_at updated_at
recipe_ingredients
Menu–ingredient join — quantity nullable for vague amounts ("a pinch of"), stores original natural language
id PK recipe_id FK ingredient_id FK quantity nullable unit quantity_display
inventory_logs
Stock in/out events — deliveries, purchases, manual adjustments; includes supplier and unit cost
id PK ingredient_id FK change_type quantity unit_cost nullable supplier nullable note nullable occurred_at
sales_logs
Per-sale records used by prediction service — recipe sold, quantity, total price, timestamp
id PK recipe_id FK quantity total_price nullable sold_at
chat_history
Multi-turn conversation history — keyed by session_id for context-aware Claude responses
id PK session_id role user|assistant content created_at

DELETE /api/recipe/{recipe_id} Delete recipe
